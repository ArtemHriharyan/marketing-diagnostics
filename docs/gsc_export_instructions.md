# Ручная выгрузка Google Search Console — инструкция для оператора

Относится к `sources.gsc.mode: "manual"` (см. `clients/<name>/config.yaml`).
Экстрактор — `src/extract/gsc_manual.py`.

## Куда класть файлы

```
inputs/manual_exports/gsc/
  2026-05/
    Запросы.csv     — обязательно
    Диаграмма.csv   — обязательно
    Страницы.csv    — обязательно, КРОМЕ комбинированного формата (см. ниже)
    Устройства.csv  — опционально
    Страны.csv      — опционально
    Фильтры.csv     — опционально
  2026-06/
    ...
```

Одна папка `YYYY-MM` = один календарный месяц. Для 12 месяцев истории — 12 папок.

## Комбинированный формат Запросы.csv (contract 3A) — предпочтительный вариант

По умолчанию отчёт Google Search Console «Эффективность» показывает данные
только по одному измерению за раз (либо запросы, либо страницы, либо
устройства). Экспорт в этом режиме теряет пересечение query×page×device —
парсер такой файл всё ещё принимает, но помечает месяц как
`incomplete_dimensions=true` (page/device в canonical будут пустыми/`unknown`).

Чтобы выгрузить сразу все три измерения в одной строке (`query, page, device`
одновременно — как их видит `gsc_api.py` при работе через API), нужно **до
экспорта** включить несколько измерений в интерфейсе:

1. Откройте **Search Console → Эффективность (Performance) → Результаты
   поиска**.
2. Задайте диапазон дат ровно на один календарный месяц (кнопка с датами
   вверху → «Настраиваемый» → 1-е число месяца — последнее число месяца).
3. Над таблицей результатов включите **все нужные вкладки-измерения
   одновременно**: «Запросы» (Queries) + «Страницы» (Pages) + «Устройства»
   (Devices). В интерфейсе GSC несколько активных вкладок-измерений
   переключают таблицу снизу в комбинированный режим — в каждой строке
   таблицы появляются все выбранные измерения сразу (запрос + страница +
   устройство), а не агрегат по одному из них.
4. Нажмите **«Экспорт» → «Скачать CSV»** (кнопка справа над таблицей).
   Google отдаёт ZIP-архив с несколькими файлами среза — используйте тот,
   что соответствует активной комбинированной таблице (обычно с колонками
   «Запрос», «Страница», «Устройство», «Клики», «Показы», «CTR», «Позиция»
   одновременно).
5. Переименуйте этот файл в `Запросы.csv` и положите в
   `inputs/manual_exports/gsc/YYYY-MM/`.
6. Отдельно, тем же способом (без переключения на комбинированный режим),
   выгрузите вкладку «Даты» → сохраните как `Диаграмма.csv` — это отдельный
   срез (временной ряд по дням), он нужен для сверки суммы кликов и не
   входит в комбинированную таблицу query×page×device.

**Если Запросы.csv собран так** (содержит колонки и Page, и Device
одновременно, под заголовками из `column_map` в конфиге клиента) —
`Страницы.csv` и `Устройства.csv` становится **не обязательно** класть
отдельно: страница и устройство уже приходят построчно из `Запросы.csv`.
Их всё ещё можно положить дополнительно (парсер использует их только для
вспомогательных `gsc_pages_YYYY-MM` / `gsc_devices_YYYY-MM`, ни на что
другое не влияет).

## Старый (раздельный) формат — всё ещё поддерживается

Если в интерфейсе доступно только одно измерение за раз (старые аккаунты,
ограничения интерфейса) — экспортируйте как раньше: `Запросы.csv` (только
query), `Страницы.csv` (только page), `Устройства.csv` (только device),
`Диаграмма.csv` (только даты). Парсер такой набор **не отклоняет**, но:

- `page=""` и `device="unknown"` для всех строк этого месяца в canonical;
- месяц попадает в `incomplete_dimensions_months` / `device_missing_months`
  (в `manifest.json` и `validation_report.json`);
- находки S08–S10, S20 (посегментный разрез по странице/устройству) для
  этого месяца не должны подниматься выше MED/LOW — данных недостаточно.

`Страницы.csv` в этом (раздельном) режиме по-прежнему обязателен — без него
месяц будет пропущен целиком с caveat `missing_required_files`.

## Сверка кликов (не меняется)

Сумма `clicks` по строкам `Запросы.csv` сверяется с суммой `clicks` по
`Диаграмма.csv` за тот же месяц. Расхождение >10% → caveat
`clicks_diagram_vs_queries_mismatch` в `caveats`/`manifest.json`. Это
отдельная проверка от `incomplete_dimensions` и не отменяет её.

## column_map

Заголовки колонок CSV зависят от языка интерфейса GSC у конкретного клиента.
Соответствие «канонический ключ → реальный заголовок» задаётся в
`clients/<name>/config.yaml` → `sources.gsc.column_map`. Для комбинированного
формата важны в первую очередь ключи `query`, `page`, `device` — именно по
наличию заголовков `column_map["page"]` и `column_map["device"]` в
`Запросы.csv` парсер определяет, комбинированный это файл или старый
раздельный.
