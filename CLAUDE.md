# marketing-diagnostics

Пайплайн диагностики маркетинга для малого бизнеса. Один аналитик, запуск
локально, Python 3.11+.

## Что это

Детерминированный конвейер, который собирает данные из рекламных/аналитических
источников (Яндекс.Метрика, Директ, Вебмастер, GSC, Wordstat, CRM), приводит их
к каноническому виду, считает метрики по реестру проверок и готовит черновики
находок для ручной проверки аналитиком перед сборкой отчёта.

## Архитектурные принципы (НЕ НАРУШАТЬ)

1. **Всё клиентское — в конфиге, не в коде.** Скрипты универсальны и читают
   `clients/<name>/config.yaml`. Новый клиент = новый конфиг, ноль правок кода.

2. **Слои конвейера неизменяемы:**
   `raw -> canonical -> metrics -> findings -> report`.
   Каждый этап читает выход предыдущего, пишет свой, ничего чужого не
   перезаписывает. Перезапись *своего* слоя целиком допустима (идемпотентность).

3. **В слоях extract/transform/compute — НИ ОДНОГО вызова LLM.** Только
   детерминированный код. LLM появляется только в слое `analyze`.

4. **Пайплайн не падает от отсутствия источника.** Каждый источник опционален,
   деградация — управляемая (см. `src/pipeline/degradation.py` и
   `config/methodology.yaml` → `requires` / `degrades_to`).

5. **Хранение:** канонические данные — parquet (pyarrow). Запросы — DuckDB
   поверх parquet-файлов (без сервера).
   *Заметка на будущее:* если появится Postgres, добавляется только модуль
   `src/storage/loader.py` — parquet остаётся источником истины.

6. **Секреты:** только в `clients/<name>/.env`. Папка `clients/` в `.gitignore`
   целиком, кроме `clients/_template/`. Никаких токенов в коде, логах и коммитах.

7. **Деньги:** суммы в рублях, integer-копейки не нужны — float с округлением
   до рубля на выводе (`config/defaults.yaml: currency_round`). Все проценты
   хранятся в долях (`0.209`), форматирование — только на выводе.

8. **Язык:** код и идентификаторы — английский; комментарии, докстринги,
   тексты находок — русский.

## Слои конвейера

| Слой        | Вход                              | Выход                                  | LLM |
|-------------|-----------------------------------|----------------------------------------|-----|
| intake      | config.yaml + .env                | таблица «источник -> доступен/нет»     | нет |
| extract     | внешние API                       | `data/raw/<source>/` + `manifest.json` | нет |
| transform   | `data/raw/`                       | `data/canonical/*.parquet`             | нет |
| compute     | `data/canonical/`                 | `data/metrics/` + `degradation_report` | нет |
| analyze     | `data/metrics/` + `inputs/`       | `findings/draft/*.yaml`                | ДА  |
| report      | `findings/approved/` + degradation| `report/`                              | нет |

## Схема ID проверок

Рабочий ID проверки — **буквенно-числовой из каталога угроз v2** (`D/A/T/C/S`
+ номер: `D01`, `A07`, `T02`, `C06`, `S03`…). Это единственный ID, который
участвует в `compute`, `degradation` и `analyze`.

Старые числовые ID предыдущей методологии (`0.1`, `1.1`, `2.1`…) хранятся в
поле `legacy_id` каждой записи `config/methodology.yaml`. Они используются
**только для сверки с отчётом Pognali v2** и НЕ участвуют в новых вызовах
`compute` / `degradation` / `analyze`. Для проверок без старого аналога
`legacy_id: null`.

Инварианты реестра (проверяются тестом): ни один `id` не встречается дважды;
ни один `legacy_id` не указывает на два разных `id` одновременно.

### Соответствие блоков

| Блок | Префикс ID | Тема                                        | Кол-во | Каталог v2 |
|:----:|:----------:|---------------------------------------------|:------:|:----------:|
| 0    | D01–D12    | Минимальное доверие к данным                | 12     | §5         |
| 1    | A01–A26    | Экономика и эффективность платной рекламы   | 26     | §6         |
| 2    | T01–T10    | Трафик, каналы и атрибуция                  | 10     | §7         |
| 3    | C01–C25    | CRO, сайт и воронка до обращения            | 25     | §8         |
| 4    | S01–S27    | SEO и органический спрос                    | 27     | §9         |

Каждая запись реестра несёт `type_default` (уровень проверяемости
`A|B|Q|A+B|A+Q`) и опциональное пост-хок понижение `type_downgrade_if` /
`type_downgraded` (условие читает булевы флаги из `data/raw/*/manifest.json`).
Итоговый `type_effective`, `source_modes` и `confidence_cap` считает
`src/pipeline/degradation.py`.

## Гейты

- `report` **запрещено** запускать, если `findings/approved/` пуст. Аналитик
  обязан проверить черновики в `findings/draft/` и переместить утверждённые
  в `findings/approved/`.
- `all` идёт последовательно и останавливается на гейтах.

## Запуск

```
python run.py <client> --stage <intake|extract|transform|compute|analyze|report|all>
```

Каждый этап пишет лог в `clients/<name>/logs/<stage>_<timestamp>.log`.

## Структура каталогов

```
config/            общие пороги (defaults.yaml) и реестр проверок (methodology.yaml)
src/pipeline/      оркестрация, манифест, карта деградации
src/extract/       по одному модулю на источник (заглушки)
src/transform/     raw -> canonical
src/compute/       block0..block6 — детерминированный расчёт метрик
src/analyze/       черновики находок (единственный слой с LLM)
src/report/        сборка отчёта
clients/_template/ полный шаблон клиента; копируется в clients/<name>/
```

## Уверенность находок

- `client-HIGH` — факт со слов клиента (`inputs/client_answers.yaml: client_facts`).
- `MED` — выводы аналитика из Вебвизора по умолчанию.
- Порог значимости и минимальная выборка — в `config/defaults.yaml`.

Уровни для находок из данных:
- `HIGH` — визит-уровень, выборка >= `min_sample_visits`.
- `MED` — ниже порога выборки либо косвенные данные.
- `LOW` — гипотеза.

Слой `analyze` может только **понижать** уверенность, никогда не повышать.
